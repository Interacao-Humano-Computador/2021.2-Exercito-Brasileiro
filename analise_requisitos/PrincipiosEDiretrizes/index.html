    Princípios e Diretrizes

    1. Introdução

    Os princípios e diretrizes de um projeto definem os objetivos gerais de mais alto nível e também regras gerais comumente vistas na prática. Norman (1998) destaca a importância e a necessidade de se projetar um sistema de forma que o usuário possa aprender rapidamente e sem muito esforço. Além disso, ele cita que o modelo conceitual deve ajudar na interpretação do relacionamento entre ações e informações apresentadas pelo sistema. Esse documento visa passar pelos princípios e diretrizes mais utilizados em IHC.

    2. Princípios e Diretrizes Gerais

    2.1 Correspondência com as Expectativas dos Usuários

    O usuário tem uma expectativa que pode ser boa ou ruim sobre o sistema. Nesse sistema a sequência de ações devem ser o mais fidedigna possível para o usuário.     2.2 Simplicidade nas Estruturas das Tarefas

    Norman (1998) recomenda deixar a estrutura de tarefas o mais simples possível, reduzindo ao máximo a quantidade de planejamento e resolução de problemas requerem. Para simplificar estruturas de tarefas complexas, Norman indica aos designers a seguir 4 abordagens tecnológicas:

    a) manter a tarefa a mesma, mas fornecendo diversas formas de apoio para que os usuários consigam aprender e realizar a tarefa;

    b) usar tecnologia para tornar visível o que seria invisível, melhorando o feedback e a capacidade de o usuário se manter no controle da tarefa;

    c) automatizar a tarefa ou parte dela, mantendo-a igual;

    d) modificar a natureza da tarefa

    2.3 Consistência e Padronização

    Facilitar o aprendizado do uso do sistema é recomendado, para isso a interface deve ser consistente e padronizada. A consistência mais importante é com as expectativas do usuário (Tognazzini, 2003), é recomendável padronizar ações, resultados das ações, layout dos diálogos e as visualizações de informação.

    2.4 Promovendo a Eficiência do Usuário

    É recomendável sempre considerar a eficiência do usuário em primeiro lugar, e não a do computador (TOGAZZINI, 2003). Pessoas são mais custosas do que máquinas, promover uma economia de tempo e esforço do usuário trazem mais benefícios do que economias semelhantes de processamento ou armazenamento. O sistema deve ser sensível ao que o usuário está fazendo e não deve interrompê-lo desnecessariamente enquanto o usuário estiver trabalhando em algo. O sistema também deve se lembrar de tudo o que o usuário disse, para não perguntar de novo, e se manter informado sobre o usuário.(COOPER, 1999)

    2.5 Antecipação

      Em todo sistema existe um padrão de ações que o usuário irá realizar. Ao projetar um sistema e uma interface, é sempre importante prever quais serão os próximos passos a ser tomados dentro de uma tarefa, para que assim o usuário possua mais informações do que pode ser realizado ou não, diminuindo assim o tempo tomado para realizar ações e poupando o usuário de procurar funcionalidades, que muitas das vezes ele pode nem saber que existe.

    2.6 Visibilidade e Reconhecimento

      Os designers de uma aplicação devem deixar, para toda ação possível, facilmente identificável o que pode ser realizado ali e como ser realizado. É importante que no desenvolvimento da interface, o designer saiba exatamente a necessidade do usuário relacionada aquela ação, identificando também a melhor maneira dela ser executada.

      De acordo com Nielsen, o designer deve explicar o que é possível de ser realizado, como é feito e também deve restringir as possibilidades de execução. Isso é importante para que o usuário não seja obrigado a memorizar como determinadas ações devem ser executadas.

    O site do exército brasileiro não possui grandes funcionalidades para exemplificar neste tópico.

    2.7 Conteúdo Relevante e Expressão Adequada

      O conteúdo de uma avaliação deve ser criado de acordo com o público alvo dela, sendo o mais claro possível para todos que irão utilizar. O uso de termos que apenas parte dos usuários conhece, pode gerar erros em determinadas ações e incompreensões na usabilidade. As expressões utilizadas devem ser claras, evitando sempre a ambiguidade, para que os usuários entendam exatamente sobre o que se trata aquela funcionalidade. As informações contidas também devem ser relevantes, sempre relacionadas com os tópicos do conteúdo. Por último, também é necessário lembrar que a informação publicada deve ter alguma relevância com o tópico, pois o excesso de informações pode trazer confusão, ou seja, as informações só devem existir na aplicação se elas realmente forem necessárias.

    2.8 Projeto para Erros

      Erros acontecem em todas as aplicações e a grande maioria deles, são por parte dos usuários. Sabendo disso, é necessário que as aplicações prevejam onde o usuário poderá cometer algum erro e fornecer, nestas etapas algum tipo de dica, sugestão ou alternativa, para que o erro seja mitigado. É importante lembrar que essas informações devem ser expostas da maneira mais clara possível, para evitar que o usuário fique confuso ou gaste bastante tempo procurando por elas. É sempre recomendado que essas dicas sejam mostradas antes mesmo do usuário realizar alguma ação.

      Como dito anteriormente, o usuário é o maior causador de erros, o que implica que a aplicação deve ser preparada para erros, fazendo com que estes sejam reversíveis. Uma aplicação que não fornece saída para erros, faz com que o usuário fique preocupado com cada ação que irá tomar, aumentando assim o tempo que ele precisa para tomar uma decisão, consequentemente o tempo necessário para realizar uma tarefa aumenta.

    3 Conclusão

      Com o que foi estabelecido nesse documento, fica claro que o sistema deve ser pensado para, ter grande coordenação entre sistema e usuário. Aplicações devem garantir facilidade no uso, ser agradáveis e o mais importante, se adaptar ao seu público alvo, sendo coerente com o tipo de usuário específico daquele sistema.

      Conforme mostrado anteriormente, o site do exército brasileiro comete alguns erros levando em conta os princípios abordados aqui.
